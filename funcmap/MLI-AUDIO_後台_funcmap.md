# MLI-AUDIO｜內勤人員審核後台 Functional Map

**專案：** MLI 高齡錄音系統輔助宣讀專案
**系統名稱：** 內勤人員審核後台
**使用者：** 內勤人員（一般審核人員、審核主管、帳號管理員、系統管理員）
**入口：** 網頁版（公司內網）
**使用情境：** 收到業務員錄製的音檔後，進行播放、STT 比對審核、案件管理與行政作業
**產出日期：** 2026-05-21

> **架構說明：** 後台為獨立網頁應用，與業務員前台共用同一後端服務、同一資料庫。
> 系統串接（建議書系統、行動投保系統、BPM、行銷系統）均由後端統一處理。
>
> **重疊項目標示：** 標記 ※ 的功能同時出現於前台文件，兩端皆需實作但視角不同。

## 模組：B-01 案件管理作業

內勤人員在後台的日常案件查詢、審查、操作與資料匯出。

| 子功能代號 | 子功能名稱 | 說明 | 優先級 |
|-----------|-----------|------|--------|
| F-013 | 手動上傳音檔 | 內勤人員可手動上傳音檔及案件欄位資訊（補件或特殊案件使用），上傳後系統自動觸發 STT 處理（F-014） | Must-have |
| F-018-1 | 地區篩選案件清單 | 依人員角色設定的所屬地區自動過濾，僅顯示對應地區的案件，跨地區人員不可見他區資料 | Must-have |
| F-018-2 | 案件搜尋 | 可依保單號碼、錄音編號、日期區間等多條件搜尋案件，支援簡易快速查詢 | Must-have |
| F-018-3 | 案件清單多功能操作 | 清單顯示案件基本資訊欄位（保單號碼、姓名、身分證字號等），操作入口包含：編輯（修改開放欄位）、照會（通知業務員回聽原音檔）、備註（系統給予案件備註）、說明（審核人員文字說明）、檢視（查看 STT 與文稿比對報告）、下載音檔（有權限者） | Must-have |
| F-021 | 案件清單匯出 | 依時間區間匯出含案件各項欄位資訊的清單檔案，供行政統計使用 | Nice-to-have |

## 模組：B-02 語音分析引擎

後端的 TTS 生成管理、STT 語音轉文字、語者辨識與原文稿比對分析。
處理結果（比對報告）供內勤人員在後台查閱與審核，亦作為輸出至 BPM 系統的依據。

| 子功能代號 | 子功能名稱 | 說明 | 優先級 |
|-----------|-----------|------|--------|
| F-006 ※ | TTS 分段音檔生成管理 | 後端收到建議書文稿後，依題號切段預先生成 TTS 音檔並管理其生成狀態、快取與清除觸發。※ 前台負責取用與播放，後台負責生成與生命週期管理 | Must-have |
| F-009 | 語者辨識 | 錄音上傳後透過第三方語者辨識服務，辨識並標示每句說話者角色（最多 6 人：機器人、業務員 1/2、要保人、被保險人、繳款人），結果整合至 STT 比對報告 | Nice-to-have |
| F-010 | STT 比對與標示 | 錄音上傳後觸發 STT 轉文字，與原文稿進行比對，標示說話人、差異處及否定語意，產出結構化比對報告供內勤人員審查確認音檔正確性 | Must-have |
| F-014 | 手動上傳觸發 STT | 內勤人員手動上傳音檔（F-013）後，系統自動觸發 STT 處理流程，共用 F-010 的 STT 管線產出比對結果 | Must-have |

---

## 模組：B-03 音檔生命週期管理

音檔的保存政策、自動清除規則、手動延長期限，以及重新合併等系統維運作業。

| 子功能代號 | 子功能名稱 | 說明 | 優先級 |
|-----------|-----------|------|--------|
| F-007 | 分段音檔條件清除 | 系統依設定條件自動清除暫存分段音檔：條件一為業務員端上傳完成後，條件二為案件有核保結果後；清除後僅保留合併音檔 | Must-have |
| F-008 | 音檔保存期限管理 | 未承保案件音檔保存五年後自動刪除；系統管理員可針對特定案件手動延長保存期限（供爭議案件持續留存使用） | Must-have |
| F-022 | 重新合併音檔 | 具有特定權限的內勤人員可針對已上傳的案件分題音檔重新執行合併作業（適用於原合併結果有誤的情況） | Nice-to-have |

---

## 模組：B-04 外部系統整合服務

後台與 BPM 系統、行銷系統的 API 串接，以及整合基礎框架。
所有外部系統串接均由後端執行，前後台共用同一整合服務。

| 子功能代號 | 子功能名稱 | 說明 | 優先級 |
|-----------|-----------|------|--------|
| F-001 ※ | BPM / 行銷系統串接基礎 | 後端與 BPM 系統、行銷系統的整合框架，負責傳遞比對結果檔及回應錄音編號清單查詢。※ 前台另與建議書系統 / 行動投保系統串接，共用同一整合服務框架 | Must-have |
| F-011 | 比對結果輸出至 BPM | STT 比對結果轉為 PDF 或 tif 格式，連同案件資訊參數一併傳送至 BPM 系統，供核保流程使用 | Must-have |
| F-012 | 行銷系統串接 | 提供 API 供行銷系統查詢符合條件之錄音編號清單（例：依日期區間、案件狀態篩選） | Must-have |

---

## 模組：B-05 存取控制與稽核管理

帳號認證、角色權限設定、安全防護與操作日誌，為後台的安全基礎設施，橫切所有模組。

| 子功能代號 | 子功能名稱 | 說明 | 優先級 |
|-----------|-----------|------|--------|
| F-016 | AD 帳密整合 | 後台登入串接公司 Active Directory，內勤人員使用既有公司帳號登入，無需另設獨立帳號 | Must-have |
| F-017 | 角色權限管理 | 可設定各人員的角色權限，細粒度控管包含：所屬地區、聽取音檔、下載音檔、新增人員帳號、重新合併音檔、手動延長保存期限等 | Must-have |
| F-019 | 操作 Log 紀錄 | 後台所有操作（查詢、異動、下載、照會、備註等）均自動記錄：操作人員帳號、執行動作、異動內容、時間（精確至日期-時-分-秒） | Must-have |
| F-020 | 閒置自動登出 | 後台靜置超過 30 分鐘自動登出，防止未授權人員取得已登入的操作權限 | Must-have |

## 待確認清單（後台相關）

| # | 問題 | 建議釐清方式 |
|---|------|------------|
| A | 語者辨識（F-009）可接受的準確率門檻？若辨識率不足，業務流程如何 fallback（人工標記）？ | 業務單位確認可接受準確率（建議 ≥85%），定義失敗時的處理規則 |
| B | STT 否定語意的定義邊界？「不一定」「不太確定」等模糊語意是否算否定？ | 業務單位提供否定語意範例語料（至少 20 句）作為 NLP 規則基礎 |
| C | 分段音檔清除的兩個條件（上傳完成後 / 核保結果後）是否需後台可設定？還是系統固定邏輯？ | 確認是否需要 B-05 提供管理員設定介面 |
| D | 「照會」功能（F-018-3）的完整流程？誰可發起、業務員如何收到通知、照會是否有期限設定？ | 補充照會流程文件，或安排業務流程訪談 |
| E | BPM 接收端偏好 PDF 或 tif？兩種格式是否均需支援，或只擇一？ | 與 BPM 廠商確認接收規格 |
| F | 各系統串接是否有統一 API Gateway，或為點對點整合？ | 與 IT 架構確認，影響 B-04 實作複雜度與維護成本 |
| G | F-019 Log 的實作方式是否採 AOP（橫切）統一記錄？需在架構設計時確認，避免各模組各自實作 Log 邏輯 | 架構設計階段確認技術方案 |
